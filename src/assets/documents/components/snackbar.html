Snackbar
Snackbars provide brief feedback about an operation through a message - typically at the bottom of the screen. They appear temporarily, towards the bottom of the screen. They shouldn’t interrupt the user experience, and they don’t require user input to disappear. They may contain a text action, but no icons. Only one snackbar may be displayed at a time.
Examples Examples
Snackbars contain a text label that directly relates to the process being performed. On mobile, the text label can contain up to two lines of text.
Props Props
property description type default
dismiss Dismisses the currently-visible snack bar.
open Opens a snackbar with a message and an optional action.
openDialogs Keeps track of the currently-open dialogs.
openFromComponent Creates and dispatches a snack bar with a custom component for the content, removing any currently opened snack bars.
openFromTemplate Creates and dispatches a snack bar with a custom template for the content, removing any currently opened snack bars.
Best Practices Best Practices
Snackbars should:
  • Informational: Snackbars provide updates on an app’s processes.
  • Temporary: Snackbars appear temporarily, and disappear on their own without requiring user input to be dismissed.
  • Contextual: Snackbars are placed in the most suitable area of the UI.
Usage & Content Guidelines Content Guidelines
On mobile, use up to two lines of text to communicate the snackbar message. In wide UIs like desktop and tablet, snackbars should have only a single line of text.
Don’t place icons in snackbars. If your message needs an icon, consider using an alert.
The text label shouldn’t share the same color as the text button.
Don’t use a filled or elevated button in a snackbar, as it draws too much attention.
